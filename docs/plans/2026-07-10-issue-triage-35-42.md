## #36 vs #42 — related, not duplicates (keep both open)

They belong to the same family — **runtime-generated transcript weight**, as opposed to the static loaded-file weight of #38–#41 — and cite the same evidence run. But they attack **different token streams** and have zero content overlap:

| | #36 trim tool output | #42 compress monologue |
|---|---|---|
| Token stream | **Input**: tool/command results (test logs, whole diffs, file reads) the orchestrator retains in its transcript | **Output**: reasoning the orchestrator + subagents *generate* each turn before their compact report |
| Fix | Redirect output to `.logs/`, read exit code + tail; diff by `--stat` + hunk | One compact-reasoning directive per charter/skill, with carve-outs for hard steps |
| Files | `workflow.md`, `build` §2, `ship` §1/§8, `wi-directory.md` — **no agent charters** | `agents/*` (all three charters) + `build`/`dev`/`rpa` skills |
| Compounding | Full ~75× (retained tool output is re-read every later turn) | Orchestrator-side: same ~75× (its own monologue stays in its transcript). Subagent-side: linear (transcripts are discarded) |
| Risk | Low — no rule content touched | Medium — over-compression can dull hard steps; needs a **real** run to validate |

They reinforce each other (less retained output → less to narrate about; less narration → less retained transcript) but neither subsumes the other. **Verdict: keep as two issues, land #36 early (Pile 1) and #42 late (careful-refactor), and never have both open against `build/SKILL.md` at the same time.** (#42's stated counterpart is actually #41 — loaded prose vs generated prose; #36 is the *third* leg: retained tool output.)

## The full taxonomy (why the order below)

1. **Measurement** — #35 (wall-clock timing). Everything else wants a before/after number.
2. **Run-state / transcript weight** (Pile 1, rule-free, safe): #36 tool output, #37 re-reads, #38 repeated reference consultation.
3. **Instruction weight** (Pile 2, moves/rewrites rule text, needs rules inventories): #39 relocate, #40 dedup, #41 compress.
4. **Generated weight** (careful, needs a real run): #42.

## Guardrails (apply per PR)

1. **Rules inventory** before/after for every PR that moves or rewords rule text (#39, #40, #41; #42's carve-out list counts). Attach it to the PR body. Test: *does each touched file still make correct decisions if loaded alone?*
2. `python scripts/validate.py` + `pytest tests/` green; after bulk markdown edits, check file **tails** (this repo has shipped mid-sentence truncated files before; `validate.py` only guards trailing-newline + fence balance).
3. Dry-run per pile; **Checkpoint B for #42 must be a real run**, not a dry-run — a simulation shares the model's blind spots.
4. New files only in already-whitelisted dirs (`references/`, `docs/` — both in `.gitignore`'s `!` list). Any *new top-level* path needs a `!/path` line or it silently vanishes.
5. `.wi/features/<slug>/.logs/` (#36) is a *user-project* path: it goes in wi-directory's strays/prune list + the scan-generated gitignore guidance, not this repo's `.gitignore`.
6. #38's resolved-routing block caches what `.wi/models.md` says — defaults stay opus/sonnet; never bake a session's top-tier model into the resolution. Settle the staleness rule at pick-up (recommend: re-resolve when the block is absent **or** `.wi/models.md` changed since it was written).
7. #35 backward compatibility: `check_tokens.py`'s new duration gate must accept legacy `tokens.md` (no Duration column) and `unavailable` values — old features' dossiers must not start failing ship.
8. No AI attribution in commits/PRs (repo convention).
